# Plan: `riboseinc/claude-memory-files` — a curated, modular, cherry-pickable repo for Claude memory & instruction files

## Context

The Ribose org wants a shared resource where developers publish portable Claude Code memory files, instruction files, and (later) hooks so colleagues can cherry-pick them into their own `~/.claude/` setups. The repo already exists at `riboseinc/claude-memory-files` with a seed issue (#1) describing the intent: start with a subset of Nick's files, then Ronald's, then Andrew2net's, then open to colleagues; structured submission process with required metadata; easy drop-in into a dev's local config; **modular cherry-pick, not monolithic adoption**, because there are files Nick explicitly objects to in Ronald's set.

Background research (conducted during planning, not tracked in-repo) found that this is a genuine white space: the entire `awesome-claude-code` ecosystem and the official plugin marketplace ship skills/agents/commands/hooks but have **no slot for memory files**. So the design isn't constrained by precedent — but three pitfalls from adjacent ecosystems must shape it:

1. **Hook RCE** (CVE-2025-59536 / CVE-2026-21852): any shared repo that ships hooks is a real supply-chain attack surface — hence v1 is memory/instructions only.
2. **Cursor-rules drift** (`PatrickJS/awesome-cursorrules`): "100% missing frontmatter, 75% too long, 50% vague" — schema without CI enforcement = noise. We enforce from day 1.
3. **Personal-style leak**: no existing schema separates "my taste" from "team practice" from "universal fact" — we add an explicit `scope:` field.

Intended outcome: a small, well-curated, CI-enforced repo with ~25 seed files, both slash-command and curl installers, Claude-mediated submission, and a clear v2 path for adding hooks once the safety review process is hardened.

**Ownership.** This is a Nick-maintained repo. Per the operative `github-narrative-location.md` convention, the design rationale and strategic decisions below are to be **posted as comments on issue #1** before any implementation work starts — not as a PR description, not as inline README copy. The plan file you are reading is the working artefact; the ticket-comment series is the canonical, browsable history.

## Approach overview

A plain git repo (not a Claude Code plugin marketplace, because plugins have no memory slot) containing:

- **Content files** under `instructions/`, `memory/`, and later `hooks/`, each with mandatory YAML frontmatter.
- **A generated `README.md`** built from frontmatter via CI, so the index never drifts.
- **Schema + linter in CI** that rejects PRs missing frontmatter fields, exceeding length caps, or (for hooks, v2) containing absolute paths / `$HOME` literals / unsigned network calls.
- **Two install paths** in parallel: a `curl | bash` one-liner per file (zero install ceremony) **and** a small companion plugin shipping `/install-memory-file` and `/remove-memory-file` slash commands with a manifest at `~/.claude/.memory-files-manifest.json`.
- **A Claude-mediated submission flow**: `/submit-memory-file <path>` reads the local file, scaffolds the PR body from frontmatter, runs the validator locally, opens the PR.

Distribution model is the **Oh My Zsh analogue**: flat-ish namespace of small files, user-side declarative manifest, opt-in by name. Not the plugin-marketplace model.

**Design notes on the layout.**

- `instructions/` vs `memory/` mirrors `~/.claude/instructions/` vs `~/.claude/memory/`. The distinction is operational: instructions get `@`-included in `~/.claude/CLAUDE.md` and **load every session** (token-costing); memory files are reference, autoloaded only via `MEMORY.md` or via an instruction file's inline reference. The installer respects this by wiring an `@`-include only for `instructions/` files.
- `_meta/index.json` is the single source of truth for the catalogue; both the slash command and the README read it. The README never has hand-written entries.
- The plugin under `tools/plugin/` is shipped via the official plugin-marketplace path; users do `/plugin marketplace add riboseinc/claude-memory-files` then `/plugin install memory-files-tools`. They are *not* installing the memory files via the plugin — they are installing the *tools* that install memory files.

## Frontmatter schema (mandatory, enforced in CI)

Every content file in `instructions/` and `memory/` starts with:

```yaml
---
schema-version: 1                      # integer; defaults to 1 if omitted. Bumped only for
                                       # breaking schema changes; additive changes are lazy.
name: kebab-case-slug                  # must match filename minus .md; unique across repo
description: One-line summary (≤140 chars). Used in README and the slash command's picker.
type: instruction
    | memory-feedback | memory-reference | memory-project | memory-user
    | settings-fragment                 # JSON snippet that DEEP-MERGES into ~/.claude/settings.json
    | project-claude-md                 # CLAUDE.md template that lands at <project>/CLAUDE.md
    | path-rule                         # .claude/rules/ file with paths: frontmatter
    | hook                              # v2 only
scope: universal | team | personal-share
                                       # universal: applies regardless of team context (e.g. github-pr-title-issue-link)
                                       # team:      a team's operating norm. May be generic team practice
                                       #            (e.g. PR-always-assigned, applicable to any team that adopts it)
                                       #            OR specific-team-only (e.g. github-no-releases for team metanorma).
                                       #            The optional `team:` field below disambiguates.
                                       # personal-share: a single author's idiosyncratic opinion shared for opt-in
                                       #            cherry-pick, NOT endorsed as a team norm (e.g. preferred narration
                                       #            tics, controversial style preferences).
team: metanorma                        # optional; only meaningful when scope: team. Names the specific team this
                                       # norm belongs to. Omit (or set to "any") for generic team practice that
                                       # any team could adopt. Used by /list-memory-files --team metanorma filter.
target:                                # where the file lands; semantics depend on type:
                                       #   instruction       → ~/.claude/instructions/<name>.md (+ @-include in CLAUDE.md)
                                       #   memory-*          → ~/.claude/memory/<name>.md (no @-include)
                                       #   settings-fragment → deep-merge into ~/.claude/settings.json (not a file copy)
                                       #   project-claude-md → <user-chosen-project-dir>/CLAUDE.md (refuses overwrite)
                                       #   path-rule         → ~/.claude/rules/<name>.md or <project>/.claude/rules/<name>.md
                                       #   hook              → ~/.claude/hooks/<name>/ (v2 only)
autoload: true | false                 # if true, installer also appends `@instructions/<name>.md` to ~/.claude/CLAUDE.md
                                       # forced true for type: instruction; forced false for memory-*, settings-fragment,
                                       # project-claude-md, path-rule (path-rule loads via paths: not @-include)
tags: [github, pr, conventions]        # 1–5 tags, drawn from a controlled vocabulary in _meta/tags.txt
author:
  name: Nick Nicholas
  github: opoudjis
owners: [opoudjis]                     # GitHub handles allowed to merge changes. Defaults to
                                       # [author.github]. For scope: personal-share, validator
                                       # enforces owners == [author.github] (length 1, exact author).
license: MIT                           # SPDX; defaults to MIT if omitted
requires:
  claude-code: ">=2.0"                 # optional; pins minimum Claude Code version
conflicts-with: []                     # list of slugs that contradict this rule; index renders a warning
supersedes: []                         # list of slugs this file replaces (for migration)
requires-companion: []                 # slugs of companion files this entry depends on (e.g. an instruction
                                       # file pointing at a data file). Installer warns when companion isn't
                                       # installed. First worked example: github-narrative-location requires
                                       # user-maintained-gems.
deprecated: false                      # tombstone flag. When true, body becomes one-paragraph stub
                                       # pointing at superseded-by; slug stays in repo as tombstone.
deprecated-reason: ""                  # required if deprecated: true
superseded-by: ""                      # optional slug; the file that replaces this one
deprecated-since: ""                   # ISO date; required if deprecated: true
changelog: []                          # optional list of {version, date, summary} entries
                                       # for minor/major bumps; aids client --update prompts
version: 1.0.0                         # semver; bumped on substantive content change. Validator
                                       # rejects PRs that change body without bumping. Patch =
                                       # typo/clarity (silent update); minor = additive (prompt
                                       # with diff); major = semantics change (explicit ack).
---

# Body (markdown, freeform)
```

**Length caps (enforced):** body ≤ 200 lines (matches Anthropic's `MEMORY.md` cap; consistent with their own CLAUDE.md guidance), description ≤ 140 chars, tags 1–5. The validator rejects PRs that exceed.

**Fields deliberately added beyond existing precedent:**
- `scope:` — the cursor-rules personal-style-leak failure mode is direct evidence this matters. `personal-share` is the legitimate slot for personal data or opinionated rules that some users will want to cherry-pick and others won't. The v1 worked example is `user-maintained-gems.md` (the data table consumed by `github-narrative-location.md`), shipped skeletonised so each user customises it. Specific-team norms (e.g. metanorma's fortnightly-release cycle in `github-no-releases.md`) sit under `scope: team` with a `team:` qualifier, not `personal-share`.
- `conflicts-with:` — multi-author repos will accumulate contradictions ("two rules contradict, Claude picks arbitrarily" is a documented Anthropic warning); we surface known conflicts in the README rather than pretending they don't exist.
- `requires-companion:` — names companion files this entry depends on. v1 worked example: `github-narrative-location.md` is universal as a mechanism but inert without its data file (`user-maintained-gems.md`); declaring the dependency lets the installer warn when the companion isn't installed.
- `owners:` — GitHub handles allowed to merge changes to this file. Drives auto-generated `CODEOWNERS`; defaults to `[author.github]`. Strict single-owner for `scope: personal-share` (validator-enforced). See "Update regulation" section below.
- `deprecated:` + `deprecated-reason:` + `superseded-by:` + `deprecated-since:` — tombstone mechanism for retraction without breaking already-installed copies. See "Update regulation" section below.
- `safety:` (v2, hooks only) — `{runs-shell, network, modifies-files, reads-credentials}` mandatory for any `hooks/` entry.

**Type-specific frontmatter requirements:**
- `settings-fragment` adds an embedded JSON block in the body inside a fenced ```json block under a `## fragment` heading. The validator parses that block and refuses entries that touch keys outside `permissions.allow`, `permissions.deny`, `permissions.ask`, or `env` in v1. Schema-validates each `allow[]` entry against the Claude Code permission-pattern grammar.
- `project-claude-md` body is the literal `CLAUDE.md` template content. Frontmatter must declare `project-type:` (e.g. `ruby-gem`, `jekyll-docs`, `rails-app`, `generic`) so the picker can filter.
- `path-rule` frontmatter must include the native Anthropic `paths:` list (this is the load-trigger; without it the file does nothing).

**Why no `paths:`/`globs:`/`alwaysApply:` in v1.** The native `.claude/rules/` `paths:` field is a Claude Code feature available for path-scoped rules; we can add it as an optional field in v1.1 once we have a real use case. Adding it speculatively complicates the schema.

<a id="bundles-and-composites"></a>
## Bundles (v1.1) and composites (v1.2) — multi-rule files with upstream tracking

A retention concern surfaced after the v1 schema settled: when several rules co-fire in the same context (e.g. four `github-*` PR-mechanics rules at every `gh pr create`), having them as separate autoloaded files dilutes attention. Users will want to merge co-firing rules into a single named hook. The v1 schema's "one rule per file" atomicity makes naïve local merges sacrifice upstream tracking — exactly the property the repo exists to provide. Full design discussion at [issue #17](https://github.com/riboseinc/claude-memory-files/issues/17); summary here.

**Bundles (v1.1).** New `type: bundle` with `includes: [slug1, slug2, ...]` frontmatter (≥ 2 entries). The bundle's body is the curated merged operational text. Constituents continue to exist as separate files upstream — installable standalone for users who only want one. The installer records both the bundle slug and each constituent's upstream ref in the manifest; `--update` against a bundle reports per-constituent divergence. Validator rules: `includes:` mandatory, each entry must resolve to an existing slug, bundle and constituents may coexist, a constituent may appear in multiple bundles. **Seed bundle for v1.1:** `github-pr-discipline` (the four `github-*` PR universals). Bundles are constrained to single-scope curations (mixing `universal` and `personal-share` upstream invites confusion about audience).

**Composites (v1.2).** Ad-hoc user-side merge. New installer mode `tools/install.sh --merge-into <local-name> <slug1> <slug2> ...` fetches each constituent's body, concatenates, writes to a user-named local file with synthetic frontmatter and a manifest `composite-of: [...]` field. `--update` walks the composite-of list; if any constituent has changed, prompts re-merge vs abort; tracks whether the local merged file has been hand-edited since merge and refuses non-interactive overwrite when it has. Composites are private to one user (not shareable); their distinguishing property versus bundles is **mixed-scope composition** — a user's lived workflow may legitimately mix `universal` rules with `personal-share` rules in one merged file. **Reference test case for v1.2:** `narration-discipline` (two universals: `narrate-plan` + `paragraph-shape`; two personal-shares: `bilingual-narration` + `bash-descriptions`). Any v1.2 PR wiring composites must demonstrate it with this cluster; until v1.2 ships, the cluster stays uncollapsed in the seed-author's local `~/.claude/` as deliberate evidence of the retention cost composites are meant to relieve.

**Why bundles before composites.** Bundles are conceptually narrower (curated, named, single-scope, shared) and so the validator/installer logic is easier to get right; composites build on the same upstream-ref-tracking infrastructure but add hand-edit-detection and re-merge semantics that benefit from a settled bundle foundation. The bundle/composite distinction is not zero-sum — both serve real use cases; they ship in sequence to contain the v1.1 surface, not because one supersedes the other.

**Seed-set impact.** v1 ships only the granular files; no bundle or composite content lands in v1. The seed file set listed in the next section is unaffected. v1.1 adds the `github-pr-discipline` bundle file (alongside the existing four constituents, all four kept standalone). v1.2 ships no new files but adds the `--merge-into` installer mode that exercises the `narration-discipline` test case end-to-end.

<a id="validation"></a>
## Validation (v1: local-only; CI deferred to v2)

**Decision: skip GitHub Actions CI in v1.** With ~25 seed files and a plausible 1–3 PRs/month from three colleagues for the first half-year, manual maintainer review during PR is adequate. The strongest single argument for CI — hook safety lint (CVE-2025-59536) — applies only to v2 when hooks land. GHA workflows also carry ongoing maintenance burden (Action pinning, deprecation chases) that isn't justified at this volume.

**What v1 ships instead:**

`_meta/validate.mjs` — a Node script (no GHA wrapper, no workflow file) that contributors run locally before opening a PR. `CONTRIBUTING.md` documents `node _meta/validate.mjs` as a mandatory pre-submission step. The script:

1. **Frontmatter validation** — reads every `*.md` under `instructions/`, `memory/`, `settings-fragments/`, `project-claude-md/`, `rules/`, parses YAML frontmatter, validates against `_meta/schema.json`. Reports: missing required field, unknown `type`/`scope`, body > 200 lines, description > 140 chars, tag not in controlled vocabulary, `name` ≠ filename, type-specific schema violations.

   **Type-specific validation:**
   - `settings-fragment`: must contain exactly one ```json``` fenced block under a `## fragment` heading. The JSON must validate as a subset of the Claude Code `settings.json` schema. The validator refuses keys outside `permissions.{allow,deny,ask}` and `env` in v1. Each `allow[]` entry is regex-validated against the permission-pattern grammar.
   - `project-claude-md`: must declare `project-type:` in frontmatter; body length capped at 200 lines (same as Anthropic's CLAUDE.md guidance).
   - `path-rule`: must declare native `paths:` list with at least one glob; each glob is syntax-checked.
2. **Conflict declaration** — if file A has `conflicts-with: [B]`, file B must either also declare A or carry an explanatory note. Warning, not error.
3. **Cross-reference validity** — `supersedes:` slugs must exist; `[[wiki-link]]` style references inside body must resolve to other files in the repo (warning).
4. **Update-regulation rules** (see "Update regulation" section for full detail):
   - PRs that change body without bumping `version:` are rejected.
   - `version:` regression or flat-on-body-change is rejected.
   - `scope: personal-share` files must have `owners == [author.github]` (length 1, exact match).
   - Deprecated files (`deprecated: true`) must have stub body + populated `deprecated-reason:` + `deprecated-since:`.
   - `_meta/build-codeowners.mjs` output is regenerated and diff-checked against committed `.github/CODEOWNERS` (force authors to commit the regenerated file).

**README is hand-written in v1.** 17 entries is well within hand-maintenance range. The `_meta/build-index.mjs` script ships (it generates `_meta/index.json` for the slash command's picker), but `_meta/build-readme.mjs` is deferred to v2. This removes the "README drift on every PR" pressure that originally motivated CI.

**v2 (when hooks land) adds GHA.** A `.github/workflows/validate.yml` is added when:
- hooks are introduced (CVE-grade safety lint earns its keep), OR
- submission volume exceeds maintainer-review capacity (subjective, but say >5 PRs/month sustained), OR
- the README outgrows hand-maintenance (~50+ entries).

The v2 GHA workflow additionally runs:
- Safety lint for hook files (no absolute paths, no `$HOME` literals, no `curl|bash`, declared `safety:` frontmatter mandatory).
- README drift check against `_meta/build-readme.mjs` output once generation is wired.
- `claude plugin validate .` on the companion plugin (Anthropic's own validator catches plugin schema regressions).

**Belt-and-braces guard against scope creep in v1:** the validator script *does* refuse PRs that add files under `hooks/` (other than `.gitkeep`). The check is in the script, runnable locally; that's enough for v1's volume.

<a id="install"></a>
## Install mechanisms (both, in parallel)

### A. `curl | bash` one-liner — zero install ceremony

Each README entry has a copy-paste line:

```bash
curl -fsSL https://raw.githubusercontent.com/riboseinc/claude-memory-files/main/tools/install.sh | bash -s -- instructions/github-pr-title-issue-link.md
```

`tools/install.sh` (POSIX, depends on `curl`, `awk`, `mkdir`, plus `jq` for `settings-fragment` merges):

1. Validates the path argument matches one of the supported subdirectories.
2. Fetches the raw file from GitHub `main` (or `--ref <sha>` for pinning).
3. Parses frontmatter to read `type`, `target`, `autoload`, `name`.
4. **Per-type install behavior:**
   - **`instruction`** — copy to `~/.claude/instructions/<name>.md`; refuse overwrite without `--force`; append `@instructions/<name>.md` to `~/.claude/CLAUDE.md` under a generated `<!-- claude-memory-files: managed block start -->` … `end` marker.
   - **`memory-*`** — copy to `~/.claude/memory/<name>.md`; no `@`-include.
   - **`settings-fragment`** — extract the embedded ```json``` block from the body; **deep-merge** into `~/.claude/settings.json` via `jq` (dedupe `permissions.allow[]` entries by exact-string match; preserve all other existing keys; refuse to touch keys outside the allowlisted set declared in `SCHEMA.md`). Back up the existing `settings.json` to `~/.claude/settings.json.bak.<timestamp>` first.
   - **`project-claude-md`** — requires a `--project <dir>` argument; copies body (frontmatter stripped) to `<dir>/CLAUDE.md`; refuses overwrite without `--force`.
   - **`path-rule`** — copies to `~/.claude/rules/<name>.md` by default, or to `<project>/.claude/rules/<name>.md` if `--project <dir>` is passed; no `@`-include needed (Anthropic's `paths:` frontmatter is the load trigger).
5. Records the install in `~/.claude/.memory-files-manifest.json` (slug, type, source ref, local path or merge-targets, install timestamp).
6. Prints a single confirmation line with what landed where and how to remove it.

A matching `tools/uninstall.sh` reads the manifest entry. For file-copy types it removes the file and strips any `@`-include. For `settings-fragment` it un-merges the recorded entries from `~/.claude/settings.json` (each entry remembered in the manifest; only those specific entries are removed; existing entries the user added by hand are left alone). For `project-claude-md` removal is manual — the manifest records the target path but the uninstaller only prints a removal hint, since the project's CLAUDE.md may have been hand-edited since install.

### B. Slash commands — managed install via a small companion plugin

Plugin at `tools/plugin/` is installed once via:

```
/plugin marketplace add riboseinc/claude-memory-files
/plugin install memory-files-tools@claude-memory-files
```

Provides four slash commands:

- `/list-memory-files [--tag <tag>] [--scope universal|team|personal-share]` — fetches `_meta/index.json`, displays a filterable picker.
- `/install-memory-file <slug-or-path>` — same effect as the curl one-liner but in-Claude. Prompts for confirmation, shows the frontmatter, asks before appending to `CLAUDE.md`. Updates the manifest.
- `/remove-memory-file <slug>` — reads manifest, removes file, strips `@`-include.
- `/submit-memory-file <local-path>` — see "Submission process" below.

The plugin itself is the only thing devs need to install once. After that, `/install-memory-file github-pr-title-issue-link` is one command and one confirmation.

**Why both, not one.** The curl path is the lowest-friction onboarding for someone who saw a single file referenced in Microsoft Teams — no plugin install required. The slash-command path is the durable workflow once a dev has bought in. Doubling the surface is justified by the different friction curves.

**Update story.** `/install-memory-file <slug> --update` re-fetches; the manifest's `source-ref` is compared against the fetched ref. Surfaced via `/list-memory-files --installed --outdated`.

<a id="submission"></a>
## Submission process (Claude-mediated)

A contributor who wants to share a file:

1. Runs `/submit-memory-file ~/.claude/instructions/my-rule.md`.
2. The slash command reads the local file, asks Claude to draft frontmatter if missing (prompting for `description`, `type`, `scope`, `tags`, `requires`). The contributor reviews and edits in-chat.
3. Slash command clones `riboseinc/claude-memory-files` into a sibling directory (per the existing `sibling-clones` convention if the user has installed that rule), creates a branch, drops the file into the correct subdirectory, regenerates the README locally, runs `_meta/validate.mjs`.
4. If validation passes, opens a PR with body assembled from the frontmatter `description` + a "Why" section the contributor fills in. Uses the existing `github-content-via-file.md` convention (PR body written to file, passed via `-F`) and the `github-pr-title-issue-link.md` convention (PR title embeds issue URL if any).
5. PR template asks the reviewer to check: does this conflict with anything existing? is the scope correct? is the body terse enough?

The submission flow is the eat-our-own-dogfood validation: the repo's own conventions (`github-content-via-file`, `github-pr-title-issue-link`, `sibling-clones`) get exercised by the act of contributing to the repo.

**Review SLA** (documented in `CONTRIBUTING.md`, not enforced in v1): aim for 2-business-day turnaround; conflicts are surfaced, not silently merged.

<a id="update-regulation"></a>
## Update regulation (v1)

A maintenance-discipline problem the seven design comments don't address: once a file lands, how do we regulate updates? Drift in installed copies, third-party amendments, deprecation, and schema migrations all matter, and the maintainer (Nick) is rightly worried about becoming a bottleneck for every typo PR.

### Author authority — `owners:` + auto-generated CODEOWNERS

Add to frontmatter:

```yaml
owners: [opoudjis]                     # GitHub handles allowed to merge changes to this file
                                       # Defaults to [author.github] if omitted.
                                       # For scope: personal-share, validator enforces
                                       # owners == [author.github] (length 1, exact author).
```

A new script `_meta/build-codeowners.mjs` walks all content files and emits `.github/CODEOWNERS` mapping each file path to its `owners:` list. Branch protection (configured by the maintainer outside tracked files):

- `scope: personal-share` → 1 review from the single named owner. Validator-enforced single-owner rule means no co-owner can dilute the signal.
- `scope: team` → 1 review from any listed owner.
- `scope: universal` → 1 review from a listed owner **plus** 1 review from the maintainer (`@opoudjis`).

**Caveat: in v1 the universal rule collapses to "Nick reviews Nick's own files"** because all 13 universal-scope seed files are authored and owned by Nick. The bottleneck relief kicks in once Ronald or Andrew author a universal-scope file: then owner = Ronald/Andrew, maintainer = Nick, and two distinct people are in the loop. The structure is in place for when contributor authorship diversifies; no instant magic for v1.

### Soft semver — tied to client `--update` behaviour

The frontmatter already has `version: 1.0.0`. Validator-enforced rules:

| Bump | What triggers it | Client `--update` behaviour |
|---|---|---|
| **patch (x.y.Z)** | Typo / wording / clarification that does not change what Claude does. | **silent** overwrite |
| **minor (x.Y.0)** | Additive behaviour change. New example, scope broadened, new optional companion. | **prompt-with-diff**, default Yes |
| **major (X.0.0)** | Semantics change, recommendation reverses, scope changes, conflicts-with newly declared, deprecation. Anything a v1-installed user might find surprising. | **require explicit ack**, default No |

Validator rejects PRs that change body without bumping `version:`; rejects `version:` regression or flat-on-body-change. Optional frontmatter `changelog: []` per entry for minor/major bumps:

```yaml
changelog:
  - version: 1.1.0
    date: 2026-05-18
    summary: Added Jekyll-4 caveat
```

### Personal-share: strict single-owner; installed copy never round-trips

Two distinct edit lanes:

- **Upstream template** (the file in the repo) — strictly the original author. The validator rejects any `scope: personal-share` PR with non-author owners. Third parties who want to amend fork as a new slug (e.g. `andrew-maintained-gems` parallel to `user-maintained-gems`), not via co-ownership of Nick's template.
- **The user's installed customised copy** — local, divergent, never round-trips. The installer sets `forked: true` in the manifest for personal-share installs at first-install time. `--update` is a no-op by default for forked entries (`--upstream` flag forces overwrite of the local copy).

This is the divergence the seed example deliberately models: Nick's metanorma row in `user-maintained-gems.md` is a worked example; every other user rewrites it on install.

### Client-side `--update` flow

Manifest extensions:

```json
{
  "slug": "github-pr-title-issue-link",
  "installed-hash": "sha256:abc...",   // body-only, frontmatter-stripped, at install time
  "upstream-ref": "<git-sha>",         // git ref of the upstream file at install
  "forked": false,                     // true for scope: personal-share
  "history": [
    { "event": "install", "version": "1.0.0", "ref": "abc123", "at": "..." },
    { "event": "update",  "version": "1.1.0", "ref": "def456", "at": "...",
      "resolution": "overwrite", "local-edited": false }
  ]
}
```

Hash-based drift detection on `--update`:

1. Fetch upstream, compute `upstream-hash`. If equal to `installed-hash` → nothing to do.
2. Compute `local-hash` (current on-disk body). Compare to `installed-hash` → was the local copy hand-edited?
3. Decision matrix:

   | Local edited? | Bump | Action |
   |---|---|---|
   | no | patch | silent overwrite; manifest history appended |
   | no | minor | show diff, default Yes |
   | no | major | show diff + changelog, default No |
   | yes | patch | prompt: keep / overwrite / diff (default keep) |
   | yes | minor or major | prompt: keep / overwrite / merge / abort (default abort) |
   | yes | personal-share, any | no-op unless `--upstream` |

Three-way merge (`git merge-file` against the recorded `upstream-ref` body as base) is deferred to v1.1; v1's safe fallback is abort.

New slash command `/list-memory-files --history <slug>` prints the history array.

### Deprecation via tombstones

Frontmatter additions:

```yaml
deprecated: true                       # default false
deprecated-reason: "Superseded by ...; original advice conflicted with ..."
superseded-by: github-pr-title-issue-link-v2   # optional slug
deprecated-since: 2026-05-18           # ISO date
```

A deprecated file's body is reduced to a one-paragraph stub pointing at `superseded-by`; `version:` bumps to major. The slug stays in the repo as a tombstone — never hard-removed, because installed copies have no other signal. `_meta/build-index.mjs` puts deprecated entries in a separate section of `index.json` so the slash-command picker hides them by default. On `--update` against an installed-but-now-deprecated slug, the installer always prompts (regardless of local-edit state) showing `deprecated-reason` and offering: keep local / auto-install successor + remove this / just remove.

A v1.1 `/check-memory-files` slash command walks the manifest and reports installed-and-deprecated files. Deferred from v1.

### Schema migrations — lazy + maintainer-side bulk

Add top-level frontmatter `schema-version: 1` (integer; defaults to 1 if omitted).

- **Additive schema changes** (new optional fields like the v1.1 `paths:`) — lazy. Files keep `schema-version: 1`; validator accepts missing optional fields. No bulk migration.
- **Breaking schema changes** (rename, type change, new required field) — ship `_meta/migrate-N-to-N+1.mjs` that opens one PR rewriting all affected files and bumps `schema-version:` across the board. Tracked in `_meta/MIGRATIONS.md`. Authors are never asked to migrate by hand.
